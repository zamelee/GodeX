## 我写了一个开源网关，让国产模型也能跑 Codex

事情的起因很简单：Codex 只认 OpenAI 的 Responses API，但 DeepSeek、MiniMax、智谱这些国产模型只提供 Chat Completions API。两个协议之间存在一条不大不小、却绕不开的鸿沟。

要么每个模型手写一套适配逻辑，维护 N 份映射代码；要么接受厂商锁定，只用一家。

两件事都不太对。

所以我写了 **GodeX**：一个跑在本地的轻量网关，你在客户端说 Responses 协议，它在背后翻译成对应模型的 Chat Completions 请求，再把结果还原成标准格式返回。对 Codex 来说，GodeX 就是一个普通的 OpenAI 兼容端点。

```text
你的 Codex / CLI / Agent
        │
        ▼
   POST /v1/responses  (OpenAI Responses API)
        │
        ▼
     GodeX (本地 Bun 进程, 端口 5678)
        │
        ▼
   DeepSeek · Xiaomi MiMo · MiniMax · 智谱
```

### 它做了什么

**模型别名路由**：把 `gpt-5.5` 映射到 `deepseek/deepseek-v4-pro`，把 `gpt-5.4-mini` 映射到 `minimax/MiniMax-M2.7-highspeed`。Codex 传什么模型名，GodeX 按配置路由，客户端零改动。想换模型？改一行 yaml。

```yaml
models:
  aliases:
    gpt-5.5: "deepseek/deepseek-v4-pro"
    gpt-5.4-mini: "minimax/MiniMax-M2.7-highspeed"
```

**能力自动降级**：如果某个模型不支持 `tool_choice: "required"` 或原生 `json_schema`，GodeX 不会悄悄忽略 — 它按 provider 的能力矩阵自动规划：能透传的透传、能降级的降级、彻底不行的提前拒绝并给出诊断信息。结构化输出降级时还会在 prompt 中注入 schema 指令，拿到结果后再校验 JSON 合法性。

**会话链 & 上下文回放**：`previous_response_id` 链式会话，每个请求的 snapshot 和 response output 都被存储，下一轮自动重建 provider 无关的历史消息。支持内存和 SQLite 两种后端，自动检测缺失父会话、循环引用和深度溢出。

**内置 Trace 可观测**：SQLite 存储的请求追踪，记录 provider 请求元数据、响应摘要、流式事件、用量数据和错误。调试时看一眼 trace 就知道上游到底返回了什么。

**流式响应**：SSE 事件流经状态机逐 delta 还原为标准 Responses 事件，中间经过 validate → trace → log → persist 的 TransformStream 管道。

### 内置四家 Provider

| Provider | 推理 | Tool Choice | 结构化输出 | 缓存 Token |
|----------|------|-------------|------------|------------|
| DeepSeek | 原生 | auto/none/required/function | json_object | ✅ |
| Xiaomi MiMo | 布尔 | auto | json_object | ✅ |
| MiniMax | 无 | auto/none/required/function | json_object | ✅ |
| 智谱 | 布尔 | auto/none | json_object | ✅ |

### 一分钟跑起来

```bash
npm install -g @ahoo-wang/godex
godex init          # 交互式选择 provider、填 API Key
godex serve         # 启动，默认端口 5678
```

然后：

```bash
curl http://localhost:5678/v1/responses \
  -H 'content-type: application/json' \
  -d '{"model":"gpt-5.5","input":"用 TypeScript 写二叉树中序遍历"}'
```

Docker 也行：

```bash
docker pull ahoowang/godex:latest
docker run -d --name godex -p 5678:5678 \
  -e DEEPSEEK_API_KEY=sk-xxx \
  -v ./godex.yaml:/etc/godex/godex.yaml:ro \
  ahoowang/godex:latest
```

### 技术选型

- **运行时**: Bun — TypeScript 原生支持，冷启动够快
- **语言**: TypeScript strict mode + ESM
- **测试**: 单元 + Mock E2E + 每家 provider 的 Live 测试
- **发布**: npm 包 + macOS/Linux/Windows 原生二进制 + Docker 镜像 (amd64/arm64)
- **协议栈**: `@ahoo-wang/fetcher` HTTP 客户端基础设施

### 设计上做了几个选择

1. **桥接逻辑与 Provider 代码严格分离**：`src/bridge/` 拥有所有 Responses-to-Chat 策略，Provider 层只暴露协议差异（spec + hooks + protocol DTOs）。加新模型不污染桥接内核，每个 provider 大约 200–400 行核心代码。

2. **没有 mapper 森林**：不搞过度抽象。先跑通，再重构。读得完也改得动。

3. **能力降级透明**：兼容性诊断信息跟随响应返回，不会让你猜「这个参数生效了吗」。

### 它是开源的

Apache-2.0 协议，欢迎 PR 和 Issue。
