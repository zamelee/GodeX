## GodeX 修复：MiniMax tool call 参数与历史配对兼容性

GodeX 在把 Codex 会话路由到 MiniMax M3 时,反复出现两类 422 错误,导致 Codex 对话无法继续:

1. `invalid function arguments json string, tool_call_id(call_function_...)` — 历史消息里 tool call 的 `arguments` JSON 字符串无法被 MiniMax 验证器接受。
2. `tool result's tool id(call_function_...) not found` — 历史消息序列里 `function_call_output` 找不到对应的 `tool_calls.id`。

这个补丁在三处加了兜底,确保发给 MiniMax 的消息序列无论上游 Codex 端如何重放历史,都是自洽的、参数是规范的。

### 根因

1. **流式累积的 `arguments` 不是 canonical JSON**。MiniMax M3 在流式输出时,`tool_calls[i].function.arguments` 字段在 SSE chunk 中可能不只发送增量,导致 GodeX state machine 字符串拼接后产生 MiniMax 验证器拒绝的格式。同时,Arguments 经过 session 重放回传给 MiniMax 时,MiniMax 期望严格的 canonical JSON。
2. **Codex 重放历史时产生 orphan tool output**。Codex 在持久化的 Responses 会话里可能包含 `function_call_output` 但缺少对应的 `function_call`(或 `tool_calls.id` 丢失/被合并吃掉),直接发到 MiniMax 会因为找不到对应 call id 而 422。
3. **无效 JSON arguments 的 tool_call 仍被透传**。即便在 step 1 修复后,如果 arguments 是彻底的非法 JSON,canonicalize 也救不了,继续发到 MiniMax 仍会 422。

### 修复

#### 1. `src/providers/shared/tool-arguments.ts` (新增)

新增两个共享工具函数:

- `canonicalizeFunctionArguments(value)` — 对 arguments 做 `JSON.parse` + `JSON.stringify`,合法时返回 canonical 形式,非法时返回原值。
- `isValidFunctionArguments(value)` — 判定 arguments 是否是合法 JSON。

#### 2. `src/providers/minimax/hooks.ts`

`minimaxPatchRequest` 在把请求发到 MiniMax 之前,新增 `canonicalizeMessageToolArguments()`:对 assistant 消息中每个 `tool_calls[i].function.arguments` 做 canonical 化。非 function 类型的 tool_call(如 `custom`)保持不动。

这是**第一道防线** — 解决 `invalid function arguments json string` 错误。

#### 3. `src/bridge/request/input-normalizer.ts`

新增 `dropUnpairedToolCalls()`,在 `normalizeResponseItems` 入口过滤 session 历史里的 unpaired `function_call` / `shell_call` / `local_shell_call` / `apply_patch_call` / `custom_tool_call`(只对当前 input items 内部的 call/output 做配对校验)。

同时 `assistantToolCallMessage()` 增加 JSON 合法性检查:5 个 tool_call 类型(`function_call` / `shell_call` / `local_shell_call` / `apply_patch_call` / `custom_tool_call`)在 arguments 不是合法 JSON 时返回 `undefined`,调用点改为 `call ? [call] : []`,**丢弃**损坏的 tool_call 而不让它继续污染对话。

这是**第二道防线** — 解决 `invalid function arguments json string` 在 provider hook 也没救回来的情况。

#### 4. `src/bridge/request/request-builder.ts`

新增 `dropOrphanToolOutputs()`,在 `buildChatMessages` 调用之前对 normalized messages 做配对校验:扫描所有 `assistant.tool_calls[*].id` 收集到 `knownCallIds`,然后丢弃 `role: "tool"` 但 `tool_call_id` 不在集合中的 message。

这是**第三道防线** — 解决 Codex 重放历史时产生的 orphan tool output。

### 测试

新增 16 个单元测试,全部通过:

- `src/providers/shared/tool-arguments.test.ts` (8 个) — 覆盖 canonicalize / isValid 的正常/边界/无效场景。
- `src/providers/minimax/hooks.test.ts` (2 个) — 验证 `minimaxPatchRequest` 对 assistant tool_call arguments 做 canonical 化,并保留非 function 类型不变。
- `src/bridge/request/input-normalizer.test.ts` (4 个) — 覆盖 5 个 tool_call 类型的 arguments 校验、function_call_output 在 function_call 被丢弃时仍保留,以及 session 入口的 call/output 配对过滤。
- `src/bridge/request/request-builder.test.ts` (2 个) — 验证 message-builder 入口的 orphan tool output 过滤。

完整测试套件:730 pass / 5 fail(5 个 fail 是 npm install 引入的环境问题,与本修复无关)。

### 影响范围

- 仅影响 MiniMax provider 的 tool call 处理路径。
- 不改变 GodeX 的对外协议、不影响其他 provider(DeepSeek / 智谱 / Xiaomi)。
- 不改变 session 存储格式、不改变 trace 记录。
- 改动的文件:
  - `src/providers/shared/tool-arguments.ts` (新增)
  - `src/providers/shared/tool-arguments.test.ts` (新增)
  - `src/providers/shared/index.ts` (re-export)
  - `src/providers/minimax/hooks.ts` (canonicalize in patchRequest)
  - `src/providers/minimax/hooks.test.ts` (新增测试)
  - `src/bridge/request/input-normalizer.ts` (drop unpaired + JSON 校验)
  - `src/bridge/request/input-normalizer.test.ts` (新增测试)
  - `src/bridge/request/request-builder.ts` (drop orphan tool outputs)
  - `src/bridge/request/request-builder.test.ts` (新增测试)

## 追加修复：MiniMax 流式 tool call 续传 null 字段兼容性

GodeX 在 Codex 持续对话过程中, 出现新的 `BridgeError: stream disconnected before completion: Provider stream delta toolCall.id must be a string when present.` 错误, 流被中断.

### 根因

MiniMax M3 的 Chat Completions **流式 tool call 续传 chunk** 与 OpenAI 标准不一致:

- OpenAI 标准: 续传 chunk (`{"index": 0, "function": {"arguments": "..."}}`) 不应携带 `id` / `name` 等字段 (字段省略).
- MiniMax 实际: 续传 chunk 把 `id` / `name` / `type` 显式设为 `null`, 只在 `function.arguments` 里续传增量. 形如:
  ```json
  {"function":{"arguments":"{\"command\": ","name":null},"id":null,"index":0,"type":"function"}
  ```

`src/providers/shared/stream-delta-mapper.ts` 的 `mapCommonChatStreamDelta` 用 `value !== undefined` 过滤字段, 但 `null !== undefined` 是 true, 因此 `id: null`、`name: null` 被原样透传到 bridge. Bridge 的 `validateToolCall` 看到 `typeof null !== "string"` 抛 `toolCall.id must be a string when present.`, 整个 SSE 流随即断开.

### 修复

**文件:** `src/providers/shared/stream-delta-mapper.ts`

把 mapper 中所有 `!== undefined` 改为 `!= null` (同时过滤 `null` 和 `undefined`), 并把 `ChatStreamToolCallDeltaLike` 类型从 `readonly id?: string` 放宽到 `readonly id?: string | null`. 这样 MiniMax 的 null-passthrough 字段会被当作"未设置"丢掉, bridge 永远看不到 null.

由于该 mapper 是 `minimax/hooks.ts`、`deepseek/hooks.ts`、`zhipu/hooks.ts` 共用的 shared helper, 修复对所有使用 Chat Completions 流式的 provider 都生效 (放宽而不是收紧, 不影响其他 provider 的正常路径).

### 测试

新增 3 个回归测试, 全部通过:

- `mapCommonChatStreamDelta null tolerance > drops null id/type/name/arguments like the field is absent` — 模拟 MiniMax 续传 chunk 形态, 验证只有非空字段透传.
- `mapCommonChatStreamDelta null tolerance > never forwards null id or name to the bridge` — 显式断言结果对象中 `id` 和 `name` 字段不存在.
- `mapCommonChatStreamDelta null tolerance > preserves valid string fields alongside null siblings` — 验证正常 chunk 不受影响.

完整测试套件: 735 pass / 4 fail (4 个 fail 是 pre-existing 的 npm install 环境问题, 与本修复无关).

### 影响范围

- 影响所有用 `mapCommonChatStreamDelta` 的 provider (minimax / deepseek / zhipu), 仅在流式 tool call 续传 chunk 含 null 字段时生效.
- 不改变协议、不影响 sync request 路径、不影响其他 mapper 行为.
- 改动的文件:
  - `src/providers/shared/stream-delta-mapper.ts` (`!== undefined` → `!= null`, 类型放宽到 `T | null`)
  - `src/providers/shared/stream-delta-mapper.test.ts` (3 个新测试)

## 追加修复：Codex 浏览器截图等 tool output 含图片时的拆分转发

Codex 内部浏览器截图返回的 `function_call_output` (以及类似的 `custom_tool_call_output`) 可能含 `input_image` part. GodeX bridge 之前的 `outputText()` 函数对 tool output 强制写死 `supportsImageInput: false` / `supportsVideoInput: false`, 于是图片在归一化阶段就被拒绝, 报:

```
Unsupported Responses input content type: input_image for minimax.
```

Chat Completions 的 `ToolMessageParam` 类型 (`content: string | ChatCompletionContentPartText[]`) 不允许 image part, 所以不能直接把图片塞进 tool message. 修复采用拆分策略:

- 文本部分保留为 `role: "tool"` 消息 (关联 `tool_call_id`).
- 图片/视频部分 (如有) 拆成一条 `role: "user"` 消息, 文本前缀 `[Attached media from tool result <call_id>]`, 把 Chat Completions 不允许的 media 通过 user message 转发给模型.

这样 tool 关联语义保留 (模型知道这是哪个 tool 的结果), 同时图片能到达 MiniMax 等 Chat Completions provider.

### 改动

**`src/bridge/request/input-normalizer.ts`**

- 新增 `NormalizedToolOutput` 类型: `{ text: string; extras: readonly ChatCompletionContentPart[] }`.
- `outputText()` 改为返回 `NormalizedToolOutput`, 不再强制关掉 image/video 支持. 文本 part 拼接成 `text`, 非文本 part 收集到 `extras`.
- `function_call_output` / `custom_tool_call_output` handler: 先 push tool message (`text`), 如果 `extras` 非空再 push 一条 user message.
- 新增 helper `toolExtrasUserMessage(extras, callId)`, 构造带文本前缀的 user message.

### 测试

`src/bridge/request/input-normalizer.test.ts` 新增 5 个测试:

- function_call_output 含 input_image 时拆成 3 条 message (assistant tool_call + tool 文本 + user 图片)
- function_call_output 只有文本时保持单条 tool message (回归保护)
- function_call_output 是 string 时保持单条 tool message (回归保护)
- custom_tool_call_output 含 image 时同样拆分
- `normalizeResponseItems` 从 session 历史里读到 image-bearing tool output 也正确拆分

input-normalizer 测试 16/16 通过, 完整套件 802 pass / 4 fail (4 个 pre-existing env 问题与本修复无关).

### 影响范围

- 仅影响 tool output 归一化路径, 仅在 tool output 含非文本 part (image / video) 时改变行为.
- 不影响 user message 路径 (user message 早就支持 image).
- 不影响 sync 工具 (`shell_call_output` / `local_shell_call_output` / `apply_patch_call_output`) — 这些的 content 本来就是 string.
- 改动的文件:
  - `src/bridge/request/input-normalizer.ts` (新增 `NormalizedToolOutput` 类型, 改 `outputText`, 改两个 handler, 新增 `toolExtrasUserMessage`)
  - `src/bridge/request/input-normalizer.test.ts` (5 个新测试)

## 追加修复：并行 tool call 场景下 media user message 插入位置错误

上一版的 image splitting 修复把含 `input_image` 的 `function_call_output` 拆成:
- `role: "tool"` (文本)
- `role: "user"` (图片, 前缀 `[Attached media from tool result <call_id>]`)

当 Codex 让 assistant 一次发起**多个并行 tool call** (例如同时调 `view_image` + `shell_command`) 时, 序列会变成:
```
assistant(tool_calls=[view_image, shell_command])
  -> tool(view_image, text)
  -> user(view_image, image)   <-- 插在这里
  -> tool(shell_command, text)
```

MiniMax 拒绝这种序列 (`tool call result does not follow tool call`), 因为它要求 assistant 后面紧跟所有 tool result, 不能在中间插入 user message.

### 修复

**`src/bridge/request/input-normalizer.ts`**

在 `normalizeInputItems` 末尾加 `reorderToolMediaMessages()` post-processing pass. 这个函数扫描生成的 chat messages, 把被夹在连续 tool message run 中间的 "media user message" (用 `[Attached media from tool result` 前缀识别) 收集起来, 等到 tool run 结束才追加:

```
工具前:                       工具后:
assistant                     assistant
tool(call_view)               tool(call_view)
user(media for call_view)     tool(call_shell)
tool(call_shell)              user(media for call_view)
[next]                        [next]
```

判定规则:
- 进入连续 `role: "tool"` run 时设 `inToolRun = true`
- 遇到 media user message 时, 如果在 tool run 内则 defer
- 遇到非 tool 也非 media 的消息时, flush deferred 然后正常追加
- run 结束后 flush 剩余 deferred

非 media user message (普通用户输入) 即使在 tool run 后面也不会被搬动, 不会破坏正常对话流.

### 测试

`src/bridge/request/input-normalizer.test.ts` 新增 4 个测试:

- "moves image user message out of the middle of parallel tool results" — 2 个并行 tool call, 第一个返回 image, 验证 image user message 被移到 tool run 之后
- "hoists multiple media user messages out of a tool run" — 3 个并行 tool call, 2 个返回 image, 验证所有 image user message 都被 hoist
- "does not reorder a media user message when it is not between tool results" — 单个 image tool output 后跟普通 user message, 验证 media message 保持在原位
- "keeps single-image tool output in place when no parallel tool follows" — 单个 image tool output, 没有并行 tool, 验证不重排

input-normalizer 测试 20/20 通过, 完整套件 806 pass / 4 fail (4 个 pre-existing env 问题与本修复无关).

### 影响范围

- 仅在 tool output 含 image/video 且出现并行 tool call 时改变消息顺序.
- 单个 tool call (无并行) 行为完全不变.
- 不影响非 media 的 user/assistant message 顺序.
- 改动的文件:
  - `src/bridge/request/input-normalizer.ts` (新增 `reorderToolMediaMessages` + `isMediaUserMessage`, `normalizeInputItems` 末尾调用 reorder)
  - `src/bridge/request/input-normalizer.test.ts` (4 个新测试)
