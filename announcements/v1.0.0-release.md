## GodeX 1.0 发布：让每个模型都成为 Codex 引擎

今天，我们正式发布 GodeX 1.0 —— 一个开源、轻量的 OpenAI Responses API 网关。它让你用一套标准协议，把 Codex、CLI 工具和开发者 Agent 接入 DeepSeek、MiniMax、智谱、Xiaomi MiMo 等 Chat Completions 模型，不再受单厂商锁定。

### 问题

OpenAI 的 Responses API 正在成为 Agent 生态的标准接口——Codex、OpenAI Agents SDK、以及越来越多的开发者工具都原生对接这套协议。但现实是：

- 很多优秀的模型（DeepSeek、MiniMax、智谱、Xiaomi MiMo）只提供 Chat Completions API，不兼容 Responses 协议。
- 直接对接每个厂商的原生 API 意味着维护多套适配逻辑，协议差异、参数映射、流式事件重构都是重复劳动。
- 每次切换模型，客户端代码和配置都要跟着改。

GodeX 把这些复杂性封装在一个本地服务里：你只需要一个 `godex.yaml`，它就能把 Responses 协议翻译成各厂商的 Chat Completions 调用，客户端无感知。

### GodeX 做了什么

**标准 OpenAI Responses API 端点** —— `POST /v1/responses` 完整支持同步和 SSE 流式响应，`/v1/models` 暴露配置的模型别名，`/health` 提供健康检查。

**多 Provider 桥接，能力自动降级** —— 内置 DeepSeek、Xiaomi MiMo、MiniMax、智谱四个 provider。每个 provider 声明自己的 capability：是否原生支持 reasoning、哪些 `tool_choice` 可用、能否处理 `json_schema` 等。GodeX 根据 capability 自动规划：支持的功能正常透传，不支持的功能降级或忽略，完全不可行的请求提前拒绝并给出清晰的诊断信息。

**智能模型路由** —— 通过 `models.aliases` 把 `gpt-5.5`、`gpt-5.4-mini` 等 Codex 模型名映射到 `provider/model`。你还可以用 `default_provider` 兜底未匹配的裸模型名，用 `*` 通配符捕获所有未知模型。切换模型只需改配置，客户端代码零变动。

**会话链 & 上下文回放** —— `previous_response_id` 会话链支持内存和 SQLite 两种后端。每个请求的 snapshot 和 response output 都被存储，下一轮自动重建 provider 无关的历史消息。

**结构化输出降级校验** —— 当 provider 不支持原生 `json_schema` 时，GodeX 自动降级到 `json_object`：在 prompt 中注入 schema 指令，要求 provider 输出 JSON，然后校验输出是否合法。非法 JSON 在同步模式下直接失败，流式模式下改写为 `response.failed` 终止事件。

**内置 Trace 可观测** —— Trace 默认开启，SQLite 存储，记录 provider 请求元数据、响应摘要、流式事件、用量数据和错误信息。调试时看一眼 trace 就知道上游到底返回了什么。

### 一分钟上手

```bash
# 安装
npm install -g @ahoo-wang/godex

# 交互式创建配置（会引导你选择 provider、填写 API Key）
godex init

# 启动
godex serve --config ./godex.yaml

# 调用
curl http://localhost:5678/v1/responses \
  -H 'content-type: application/json' \
  -d '{"model": "gpt-5.5", "input": "用 TypeScript 写一个二叉树的中序遍历"}'
```

Docker 用户：

```bash
docker pull ahoowang/godex:latest
docker run -d --name godex -p 5678:5678 \
  -e DEEPSEEK_API_KEY=sk-xxx \
  -v ./godex.yaml:/etc/godex/godex.yaml:ro \
  ahoowang/godex:latest
```

### 技术架构

GodeX 的核心是一个 **Responses-to-Chat 桥接内核**（`src/bridge`），它把 provider 无关的兼容性规划、请求构建、工具映射、输出契约、响应重构和流式状态机集中在一处。Provider 层只暴露协议差异（spec + hooks + protocol DTOs），不做重复的适配决策。

请求链路：`POST /v1/responses` → 解析模型别名 → 解析会话链 → 选择 ProviderEdge → Bridge 构建 Chat Completions 请求 → Provider 调用上游 → Bridge 重构 Responses 输出 → Trace / 日志 / 会话持久化。

详见 [架构文档](https://github.com/Ahoo-Wang/GodeX?tab=readme-ov-file#architecture)。

### 它是怎么构建的

- **运行时**：Bun — 冷启动快、TypeScript 原生支持、内置 test runner
- **语言**：TypeScript strict mode，ESM，verbatimModuleSyntax
- **测试**：单元测试 + Mock E2E + 各 provider 的 Live 测试
- **发布**：npm 包 + 多平台原生二进制 + Docker 镜像（linux/amd64, linux/arm64）
- **协议栈**：基于 `@ahoo-wang/fetcher` 的 HTTP 客户端基础设施

### 路线图

GodeX 1.0 是起点。接下来我们会关注：

- **更多 Provider**：欢迎社区贡献新的 provider 实现，ProviderSpec 接口已经稳定。
- **Stream 管道可扩展**：让用户自定义 TransformStream 插入 stream pipeline。
- **Admin API & Dashboard**：可视化管理模型别名、Trace 查询、用量统计。
- **配置热加载**：`godex.yaml` 变更后无需重启即可生效。

### GodeX 1.0: Make Every Model a Codex Engine

Today we're releasing GodeX 1.0, an open-source, lightweight OpenAI Responses API gateway. It lets clients that speak the Responses protocol — Codex, OpenAI Agents SDK, CLI tools — use Chat Completions providers like DeepSeek, MiniMax, Zhipu, and Xiaomi MiMo through a single local server, with no client-side provider logic.

**What it does:**

- Full `POST /v1/responses` endpoint with sync and SSE streaming
- Multi-provider bridge with automatic capability-based degradation (tools, structured output, reasoning, tool_choice)
- Smart model aliasing — map `gpt-5.5` to `deepseek/deepseek-v4-pro` in config, zero client code changes
- Session chains with memory or SQLite backends for `previous_response_id` context replay
- Structured output downgrade: `json_schema` → `json_object` with prompt injection and output validation
- Built-in SQLite tracing for requests, usage, stream events, and errors
- Bun runtime, TypeScript, cross-platform binaries, Docker images

GodeX is Apache-2.0 licensed and welcomes provider contributions. If your favorite model speaks Chat Completions but not Responses, a GodeX provider spec is all it takes.
