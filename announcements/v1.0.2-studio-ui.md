## GodeX Studio 修复:连接设置可落盘 + 多 provider / 模型 UI

GodeX Studio UI (http://127.0.0.1:56791) 之前虽然跑起来了,但几个最常用的按钮实际上是哑的:

- `/api/providers` POST 返回 `{"ok":true}` 但 config.yaml 里 **没有真的新增** provider。
- `/api/providers/:name` DELETE 一直返回 `{"ok":false,"error":"not found"}`,即使 provider 确实存在。
- `/api/config` POST (`applyConfig`) 写出来的 config.yaml 会出现 **重复的 provider 块**,导致 godex 加载时报错。
- 前端 HTML 完全没有 `<script>` 块,所有按钮 onclick 都不响应,Model 列表永远显示"加载中..."。
- 右上角 health 永远显示 `godex offline: Failed to fetch`,因为 trace DB 路径写死了 `godex-new/data/trace.db`(并不存在)。

这次补丁把连接设置、provider/model 增删、模型列表拉取、日志轮询全部串通,Studio 真正能当 GodeX 的控制面板用了。

### 根因

1. **regex literal 里 `\\s\\n\\S\\Z` 写错**。`nameEsc.replace(/[.*+?^${}()|[\\]\\\\]/g, "\\\\$&")` 和后面两段 `new RegExp("\\n[\\s\\S]*?)(?=\\n {2}[^\\s]...")` 在 Bun runtime 下,`\\s` 被解析成 regex source 里的字面 `\\s` (匹配反斜杠 + s),而不是空白字符 `\s`,所以 upsert / delete / add 三个分支全部走错路。`/api/providers` POST 的 else 分支还有一段 `raw.replace(/(providers:\\s*\\n)/, ...)` regex literal 同样问题,导致添加新 provider 完全不生效。
2. **`writeConfigYaml` 是覆盖式写入**。每次 apply 直接 `fs_write` 一份新的 yaml,丢掉 config.yaml 里所有其他 provider 块 — 切回原 provider 后它的 base_url / api_key 也丢了。
3. **trace DB 路径写错**。`TRACE_DB_PATH` 默认 `../../godex-new/data/trace.db` 是相对路径,在 `D:\\Documents\\VibeCoding\\GodeX\\platforms\\win32-x64\\bin\\` 启动时解到完全错的地方,`/api/logs` 永远返回 `[]`。
4. **HTML 没有 `<script>`**。`onclick="fetchAllModels()"` / `onclick="applyConfig()"` / `onclick="toggleAddProvider()"` 等事件全都不存在对应函数,Studio 完全不能交互。

### 修复

#### 1. `studio/src/serve.ts`

- 新增 `readAllProviders()` — 解析 config.yaml 的 `providers:` 块,返回 `[{ name, spec, base_url, api_key, timeout_ms }]`,被 `GET /api/providers` 复用。
- 新增 `upsertProvider(name, baseUrl, apiKey, timeoutMs, spec)` — 找到现有 `<name>:` 块就 replace 掉,没有就追加到 `providers:` 段尾。保留所有其他 provider 块。
- `writeConfigYaml()` 改为先 `upsertProvider` 替换 active 块,再合并 aliases,不再整体覆盖。
- `GET /api/upstream-models` 的 regex literal `/\\/+$/` (有 backslash escape 问题) 改成 char class `/[\\/]+$/`。
- 新增 `GET /api/all-models` — 并行请求所有 provider 的 `/v1/models`,返回 `[{ name, spec, models: [{ id }], error? }]`,前端可以按 provider 分组渲染。
- 修 `/api/providers` POST 的 else 分支 regex literal `/(providers:\\s*\\n)/` 中 `\\s` 改成 `\s`。
- 修 `/api/providers` POST 的 nameEsc.replace 字符类,正确处理 provider 名中的特殊字符。
- `TRACE_DB_PATH` 默认路径改为 `C:\\Users\\Bliss\\.godex\\data\\trace.db`,新增 `GODEX_TRACE_DB` env var 覆盖。
- `GODEX_CONFIG` / `GODEX_BINARY` 默认值都改成绝对路径,不再依赖 cwd。

#### 2. `studio/public/index.html`

- 末尾追加完整 `<script>...</script></body></html>`,包含:
  - 渲染左侧 provider 列表(点选 active、点 × 删除,带确认)
  - 渲染右侧 model 列表(按 provider 分组,带 filter 搜索框)
  - 中央连接设置表单(Provider / Base URL / API Key / Timeout / Spec / 各种高级参数 / alias)
  - `MODEL_PRESETS` 表 — 12 个常见 model id (MiniMax-M3 / M2.7 / M2.7-highspeed / deepseek-reasoner / o1 / o3 / o4-mini / gpt-4o / glm-4-plus 等) 的 context_window / max_output / thinking / reasoning_effort 默认值
  - 点 model 卡片自动填表单:context、max_output、thinking、reasoning_effort,以及 `alias_target = '<provider>/<model id>'`、`alias_default = '<provider>-<model slug>'`
  - "应用到 godex" 按钮:写 config.yaml + kill old godex + start new godex,完成后 6s 自动 reload 页面
  - "重启 godex" 按钮复用 apply 路径
  - "保存" 按钮把当前表单值 localStorage 持久化(`studio:config` key)
  - "Fetch URL" 弹窗:输入任意 base URL 临时拉 `/v1/models`,结果合并到右栏 model 列表,不写到 config
  - 每 5s 轮询 `/api/logs`,每 8s 轮询 `/api/health`
  - `__GODEX_BASE__` / `__STUDIO_PORT__` 由 serve.ts 在 HTML 加载时替换

### 行为变化

- 左侧选 provider → 中间自动填它的 base_url / api_key / spec / timeout
- 点右侧任意 model 卡片 → 自动套用 preset (context_window / max_output / thinking) + 设好 alias
- "应用到 godex" 写入 config.yaml 时,**保留**所有其他 provider 块,只替换 active provider
- "Fetch URL" 拉回的结果**只**显示在 UI 上,不写到 config.yaml
- 右下角日志流式显示 godex trace DB 里的 `trace_events` 行
